[Windows] DRAW is **DPI-unaware**, so on a 4K panel set to Windows display scaling (e.g. 150%), `_DESKTOPWIDTH`/`_DESKTOPHEIGHT` report the DPI-virtualized desktop (observed: **1536x864** on grymm's 3840x2160 panel), NOT the true resolution. Windows then bitmap-stretches DRAW's window up to the real panel — everything is uniformly soft/enlarged. This is why scaling looked "all over the place" on Windows but not Mac/Linux (no such virtualization).

**Two render coordinate spaces** caused the multi-scale mess:
- Main UI (canvas, toolbar, palette, menus, panels) renders into the viewport buffer and chrome blits to the window at `CFG.TOOLBAR_SCALE%` (ts); canvas magnifies by `SCRN.displayScale%` (ds = DISPLAY_SCALE).
- The **modal file dialog** blits 1:1 to page 0 ([GJ-DIALOG-SCALE.BM](GUI/GJ-DIALOG-SCALE.BM) `DRAW_FD_apply_scale_and_center` sets `FD_STATE.scale=1`) — it escapes ds and had its own `FD_FONT_SCALE`/`FD_ICON_SCALE`, plus the DEFAULT theme loads its font (Tiny5) at `FILE_DIALOG_FONT_SIZE=16` = 2x the 8px UI font. Old defaults (FD_FONT_SCALE=2, FD_ICON_SCALE=3) → ~32px dialog text = ENORMOUS.

**Two render families:** viewport-baked chrome (toolbar, menu, palette, status, layers) renders into SCRN.CANVAS& then upscales ×ds; floating overlays (color mixer, browser, modal dialogs) blit to screen 0 at ×ts. When ds≠ts (e.g. 4K: ds=4, ts=2) the two families differ in size. The PREVIEW window was originally viewport-baked (×ds), so it looked 2× the floating panels. **Fixed 2026-06:** converted PREVIEW to a floating overlay (×ts), mirroring COLORMIXER — renders into PREVIEW.imgHandle& at native size, blits to screen 0 at `(x*ds,y*ds)` size `(w*ts,h*ts)`; PREVIEW.w/h are now NATIVE, renderScale=ts/ds, dispW/dispH=footprint; input maps RAW→native via `/renderScale`; PREVIEW_render call moved from viewport section to overlay section (before COLORMIXER_render) in all 3 SCREEN_render paths; loupe/tooltip render native→×ts to screen 0. Highest-risk areas: loupe/tooltip placement and 8-way resize.

**Panel size consistency:** every panel renders its font then blits ×ts, so effective text size = fontBase × fontScale × blit. Color mixer/preview/chrome use GLOBAL_FONT @ 8px → 8·ts. The FD library (modal file dialog + docked browser) used a 16px Tiny5 base; modal blits ×1 so ts/2 compensated it, but the browser blits ×ts and integer fontScale can't be 0.5 → browser was stuck at 16·ts (2× everything). Fix: set DEFAULT theme `FILE_DIALOG_FONT_SIZE = 8` (matches GLOBAL) and modal `DRAW_FD_apply_font_scale` derive `fs = ts` (was `(ts+1)\2`). Now all panels = 8·ts. The per-panel font sources: [COLOR-MIXER.BM:97](GUI/COLOR-MIXER.BM#L97), [PREVIEW.BM:92](GUI/PREVIEW.BM#L92), FD via [GJ-DIALOG-SCALE.BM:598](GUI/GJ-DIALOG-SCALE.BM#L598). Floating panels have NO independent scale knob — color mixer/picker/msgbox follow TOOLBAR_SCALE; only browser (BROWSER_*) and file dialog (FD_*) have optional overrides (0=auto-derive from ts).

**Frozen-viewport gotcha (fixed 2026-06-07):** config save used to freeze the resolved `SCREEN_WIDTH/HEIGHT` even when `UI_SCALE` drove the layout. A viewport frozen at one scale (e.g. 960x496) then loaded under a different `UI_SCALE` (e.g. 4 → should be 960x540) made the viewport inconsistent with the scale → window/canvas misaligned, oversized chrome, stale blits ("messed up UI"); also broke the Settings dialog (centered against a mismatched viewport → mouse mis-mapped, only ESC worked). Fix: `SCREEN_init` ignores configured `SCREEN_WIDTH/HEIGHT` whenever `CFG.UI_SCALE% > 0` (always derives desktop/scale); `CONFIG_save` writes `SCREEN_WIDTH=0/HEIGHT=0` when `UI_SCALE>0`. Only pure Auto mode (UI_SCALE=0) honors a configured viewport.

**Ctrl+, opens Settings — [Windows] Ctrl+comma keycode quirk (verified via a probe):** when Ctrl is held, SDL/QB64 on this Windows PC SUPPRESSES the comma's ASCII code — `_KEYDOWN(44)` stays FALSE and the key only arrives through `_KEYHIT` as the virtual-key code **188** (VK_OEM_COMMA), as ±188 (press/release). Comma WITHOUT Ctrl is `_KEYDOWN(44)` / `_KEYHIT 44` as normal. So any Ctrl+comma handler using `_KEYDOWN(44)` (or `44 OR 60`) silently never fires here. Fix: `KEYBOARD_input_handler` checks `LAST_KEYHIT_RAW& = 188 OR -188 OR 44` (the per-frame `_KEYHIT` captured in DRAW.BAS) gated by Ctrl & NOT shift/alt. The INPUT.BM central binding for ACTION_SETTINGS is `dispatched=FALSE`. General lesson: Ctrl+punctuation on Windows/SDL must be detected via `_KEYHIT` VK codes, not `_KEYDOWN(ascii)`. A reusable probe is `DEV/keytest.bas` (logs every `_KEYDOWN`/`_KEYHIT` code to keytest.log). **UPDATE (2026-06-07, fixed on Linux):** the `_KEYHIT`-only handler above silently broke Ctrl+, on Linux/macOS — there `_KEYHIT` doesn't deliver the comma under Ctrl, and the central `_KEYDOWN(44)` dispatch path was disabled by the `dispatched=FALSE` flip. The handler now ORs BOTH: `_KEYDOWN(44) OR LAST_KEYHIT_RAW& = 188 OR -188 OR 44`, with a STATIC edge guard (since `_KEYDOWN` is level-triggered) re-armed in an outer ELSE. So Ctrl+punctuation needs BOTH detectors, not just `_KEYHIT`. See canonical note `feedback_ctrl_punctuation_cross_platform`.

**Window must not fill the desktop exactly:** explicit scale used to size the window to the full desktop (viewport=desktop/scale), but the OS title bar (large under DPI awareness) + taskbar pushed it past the screen, so Windows clipped + mis-positioned it → broke canvas layout, made dialog clicks miss (mouse coords ≠ rendered positions), and opened modal dialogs off-screen. Fix: SCREEN_init + SCREEN_set_display_scale always leave a ~10% margin (was explicit-scale-only-fills). Symptom to watch: GetWindowRect shows the window larger than / offset from the desktop.

**Fix (2026-06): one master `UI_SCALE`** (0=auto, 1-8) in `CFG.UI_SCALE%`. When >0 it sets DISPLAY_SCALE and derives TOOLBAR_SCALE via `SCREEN_chrome_scale_from_display%`. The dialog/browser scales now **auto-derive from TOOLBAR_SCALE when 0**: modal `FD_FONT_SCALE = (ts+1)\2` (compensates the 16px font base), `FD_ICON_SCALE = ts`; browser derives to 1 (it blits ×ts already). 0 = auto everywhere, persisted as 0 (never frozen). Exposed in Settings → Display as "UI Scale (master)". Also config save "freezes" auto-detected ds/viewport as explicit values on first save — a separate staleness source.

**DPI awareness (done 2026-06, tested):** [Windows] `DECLARE DYNAMIC LIBRARY "user32"` → `SetProcessDPIAware` (aliased `DRAW_SetProcessDPIAware&`) in [SCREEN.BI](OUTPUT/SCREEN.BI), called at the TOP of `SCREEN_init` right after `CONFIG_load` (before the first `_DESKTOPWIDTH`/window) gated on `CFG.HIDPI_AWARE%` (default TRUE; config key `HIDPI_AWARE=1`). Confirmed in log: `SetProcessDPIAware -> 1 (desktop now 3840x2160)` — was 1536x864 — and auto-detect then correctly picks 4K preset (ds=4, TB=2). All wrapped in `$IF WIN THEN`/`$END IF`, so Linux (real pixels) and macOS (Retina backing) compile/run unchanged with no equivalent call. Escape hatch: set `HIDPI_AWARE=0` if window/mouse positioning misbehaves. Mouse-alignment under DPI awareness is the main thing to watch on new Windows setups.
