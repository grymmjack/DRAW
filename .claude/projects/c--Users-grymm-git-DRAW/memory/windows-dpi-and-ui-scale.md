[Windows] DRAW is **DPI-unaware**, so on a 4K panel set to Windows display scaling (e.g. 150%), `_DESKTOPWIDTH`/`_DESKTOPHEIGHT` report the DPI-virtualized desktop (observed: **1536x864** on grymm's 3840x2160 panel), NOT the true resolution. Windows then bitmap-stretches DRAW's window up to the real panel ‚Äî everything is uniformly soft/enlarged. This is why scaling looked "all over the place" on Windows but not Mac/Linux (no such virtualization).

**Two render coordinate spaces** caused the multi-scale mess:
- Main UI (canvas, toolbar, palette, menus, panels) renders into the viewport buffer and chrome blits to the window at `CFG.TOOLBAR_SCALE%` (ts); canvas magnifies by `SCRN.displayScale%` (ds = DISPLAY_SCALE).
- The **modal file dialog** blits 1:1 to page 0 ([GJ-DIALOG-SCALE.BM](GUI/GJ-DIALOG-SCALE.BM) `DRAW_FD_apply_scale_and_center` sets `FD_STATE.scale=1`) ‚Äî it escapes ds and had its own `FD_FONT_SCALE`/`FD_ICON_SCALE`, plus the DEFAULT theme loads its font (Tiny5) at `FILE_DIALOG_FONT_SIZE=16` = 2x the 8px UI font. Old defaults (FD_FONT_SCALE=2, FD_ICON_SCALE=3) ‚Üí ~32px dialog text = ENORMOUS.

**Two render families:** viewport-baked chrome (toolbar, menu, palette, status, layers) renders into SCRN.CANVAS& then upscales √óds; floating overlays (color mixer, browser, modal dialogs) blit to screen 0 at √óts. When ds‚âÝts (e.g. 4K: ds=4, ts=2) the two families differ in size. The PREVIEW window was originally viewport-baked (√óds), so it looked 2√ó the floating panels. **Fixed 2026-06:** converted PREVIEW to a floating overlay (√óts), mirroring COLORMIXER ‚Äî renders into PREVIEW.imgHandle& at native size, blits to screen 0 at `(x*ds,y*ds)` size `(w*ts,h*ts)`; PREVIEW.w/h are now NATIVE, renderScale=ts/ds, dispW/dispH=footprint; input maps RAW‚Üínative via `/renderScale`; PREVIEW_render call moved from viewport section to overlay section (before COLORMIXER_render) in all 3 SCREEN_render paths; loupe/tooltip render native‚Üí√óts to screen 0. Highest-risk areas: loupe/tooltip placement and 8-way resize.

**Panel size consistency:** every panel renders its font then blits √óts, so effective text size = fontBase √ó fontScale √ó blit. Color mixer/preview/chrome use GLOBAL_FONT @ 8px ‚Üí 8¬∑ts. The FD library (modal file dialog + docked browser) used a 16px Tiny5 base; modal blits √ó1 so ts/2 compensated it, but the browser blits √óts and integer fontScale can't be 0.5 ‚Üí browser was stuck at 16¬∑ts (2√ó everything). Fix: set DEFAULT theme `FILE_DIALOG_FONT_SIZE = 8` (matches GLOBAL) and modal `DRAW_FD_apply_font_scale` derive `fs = ts` (was `(ts+1)\2`). Now all panels = 8¬∑ts. The per-panel font sources: [COLOR-MIXER.BM:97](GUI/COLOR-MIXER.BM#L97), [PREVIEW.BM:92](GUI/PREVIEW.BM#L92), FD via [GJ-DIALOG-SCALE.BM:598](GUI/GJ-DIALOG-SCALE.BM#L598). Floating panels have NO independent scale knob ‚Äî color mixer/picker/msgbox follow TOOLBAR_SCALE; only browser (BROWSER_*) and file dialog (FD_*) have optional overrides (0=auto-derive from ts).

**Fix (2026-06): one master `UI_SCALE`** (0=auto, 1-8) in `CFG.UI_SCALE%`. When >0 it sets DISPLAY_SCALE and derives TOOLBAR_SCALE via `SCREEN_chrome_scale_from_display%`. The dialog/browser scales now **auto-derive from TOOLBAR_SCALE when 0**: modal `FD_FONT_SCALE = (ts+1)\2` (compensates the 16px font base), `FD_ICON_SCALE = ts`; browser derives to 1 (it blits √óts already). 0 = auto everywhere, persisted as 0 (never frozen). Exposed in Settings ‚Üí Display as "UI Scale (master)". Also config save "freezes" auto-detected ds/viewport as explicit values on first save ‚Äî a separate staleness source.

**DPI awareness (done 2026-06, tested):** [Windows] `DECLARE DYNAMIC LIBRARY "user32"` ‚Üí `SetProcessDPIAware` (aliased `DRAW_SetProcessDPIAware&`) in [SCREEN.BI](OUTPUT/SCREEN.BI), called at the TOP of `SCREEN_init` right after `CONFIG_load` (before the first `_DESKTOPWIDTH`/window) gated on `CFG.HIDPI_AWARE%` (default TRUE; config key `HIDPI_AWARE=1`). Confirmed in log: `SetProcessDPIAware -> 1 (desktop now 3840x2160)` ‚Äî was 1536x864 ‚Äî and auto-detect then correctly picks 4K preset (ds=4, TB=2). All wrapped in `$IF WIN THEN`/`$END IF`, so Linux (real pixels) and macOS (Retina backing) compile/run unchanged with no equivalent call. Escape hatch: set `HIDPI_AWARE=0` if window/mouse positioning misbehaves. Mouse-alignment under DPI awareness is the main thing to watch on new Windows setups.
